### 18:48 - 电子宠物项目部署完成

**项目名称**: E-Pet (电子宠物)

**访问地址**: https://soa.laziestlife.com/epet/

**技术栈**:
- 后端: Express.js + PostgreSQL
- 前端: 纯 HTML/CSS/JS 单页
- 宠物形象: SVG 内嵌（橙色卡通猫）

**核心功能**:
- 首次访问自动初始化宠物（生成随机 pet_id）
- 抚摸：点击猫头或按钮，触发抖动动画
- 喂食：饥饿度回满，累计喂食次数 +1
- 铲屎：累计铲屎次数 +1
- 饥饿度衰减：每 10 分钟 -7，约 24 小时归零（低于 30% 显示红色）

**API 端点**:
- GET /api/epet/init - 初始化新宠物
- GET /api/epet/:id - 获取宠物状态
- POST /api/epet/:id/feed - 喂食
- POST /api/epet/:id/pet - 抚摸
- POST /api/epet/:id/clean - 铲屎

**服务器配置**:
- 部署路径: /opt/app/
- Nginx 代理: /epet/ -> localhost:3000/api/epet/
- 数据库: PostgreSQL (表 pets)
- 启动命令: pm2 start server-epet.js --name epet

**数据库表结构** (pets):
- pet_id (VARCHAR, 主键)
- hunger (INTEGER, 默认 100)
- total_feeds, total_poops, total_pets (INTEGER, 默认 0)
- last_feed_at, last_poop_at, last_interact_at (TIMESTAMP)
- created_at (TIMESTAMP)

**待优化**:
- 宠物形象可替换为用户提供的 PNG
- 饥饿度 0 时的新状态图
- 成就系统
### 18:59 - 电子宠物 NFC 部署逻辑更新

**业务流程变更**:
- **原计划**: 用户首次访问时后端随机生成 pet_id
- **新方案**: 预先生成一批 NFC ID，烧录到 NFC 卡片中
  - 用户扫码访问 `/epet/?id=<nfc_id>`
  - **首次访问时**，该 ID 才落库（创建宠物）
  - 后续访问同一 ID，直接加载已有宠物

**需要修改的后端逻辑**:
1. ❌ 移除 `/api/epet/init` 接口（不再需要后端生成 ID）
2. ✅ `GET /api/epet/:id` 改为：
   - 查询数据库是否存在该 ID
   - 不存在 → 自动 INSERT 创建宠物
   - 存在 → 返回宠物状态
3. ✅ 前端移除"初始化"逻辑，直接读取 URL 参数 `?id=xxx`
