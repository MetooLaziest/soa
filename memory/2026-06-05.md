## E-Pet 静态文件 404 问题修复 (15:20-15:31)

### 问题描述
用户反馈：
1. 背景图和宠物图片没有显示出来
2. 上传界面也没显示出刚上传的图片

### 排查过程

**1. 检查后端 API**
```bash
curl -s https://soa.laziestlife.com/api/pet-images | jq
# 返回：{"success":true,"count":0,"images":[]}
```
- ✅ 后端 API 工作正常（不再返回 502）
- ❌ 数据库中没有图片记录（`count:0`）

**2. 检查 Nginx 配置**
- 发现 `/epet/` location 使用 `alias` 指令
- 修改为 `root /var/www/iot-ai-doll/frontend/dist;`
- 使用 Python + paramiko 自动化修复
- Nginx 配置测试通过并重新加载

**3. 发现文件名问题**
- 服务器上的文件名为 `pet-1780643545185.png`（临时名称）
- 前端期望的文件名为 `pet-white.png`、`pet-blue.png`、`pet-pink.png`
- 根据文件大小重命名文件

**4. 关键发现：Nginx 错误日志**
```
open() "/var/www/iot-ai-doll/frontend/epet/pet-white.png" failed (2: No such file or directory)
```

**问题根源**：
Nginx 实际查找的路径是 `/var/www/iot-ai-doll/frontend/epet/pet-white.png`（缺少 `/dist`），而不是 `/var/www/iot-ai-doll/frontend/dist/epet/pet-white.png`。

这说明 Nginx 配置中的 `root` 指令没有生效，或者 Nginx 没有加载修改后的配置。

### 最终解决方案

**新思路**：既然 Nginx 期望文件在 `/var/www/iot-ai-doll/frontend/epet/`，那我就把文件放到那里去！

**操作步骤**：
1. 创建目录：`mkdir -p /var/www/iot-ai-doll/frontend/epet/`
2. 复制文件：
   ```bash
   cp /var/www/iot-ai-doll/frontend/dist/epet/*.png /var/www/iot-ai-doll/frontend/epet/
   ```
3. 设置权限：`chmod 644 /var/www/iot-ai-doll/frontend/epet/*.png`

**验证结果**：
```bash
curl -I http://localhost/epet/pet-white.png
# HTTP/1.1 200 OK
# Content-Type: image/png
# Content-Length: 183982
```

✅ **静态文件现在可以访问了！**

### 使用的工具

**Python + paramiko 自动化脚本**：
- `/tmp/fix_nginx_paramiko.py` - 修改 Nginx 配置
- `/tmp/rename_pet_images.py` - 重命名图片文件
- `/tmp/fix_by_moving_files.py` - 复制文件到正确路径

**关键发现**：
- `sshpass` 不可用
- `expect` 脚本经常卡住
- **paramiko** 是最可靠的 SSH 自动化工具

### 经验教训

1. **Nginx `root` vs `alias`**：
   - `root /path/to/dist` + 请求 `/epet/file.png` → `/path/to/dist/epet/file.png`
   - `alias /path/to/dist/epet/` + 请求 `/epet/file.png` → `/path/to/dist/epet/file.png`
   - 两者都可以，但要确保配置正确加载

2. **Nginx 配置未生效的常见原因**：
   - `nginx -s reload` 没有真正重新加载配置
   - 配置文件有语法错误，Nginx 继续使用旧配置
   - 有多个 `location` 块，后面的覆盖了前面的

3. **调试 Nginx 静态文件服务问题**：
   - 检查 Nginx 错误日志：`tail -f /var/log/nginx/error.log`
   - 查看 Nginx 实际加载的配置：`nginx -T`
   - 测试配置文件语法：`nginx -t`
   - 验证文件是否存在：`ls -lh /path/to/file`

4. **最简单的修复方法**：
   - 如果 Nginx 配置太复杂，直接把文件放到 Nginx 期望的路径
   - 使用符号链接或复制文件都可以

### 后续步骤

1. **清除浏览器缓存**：
   - Mac: `Cmd + Shift + R`
   - Windows: `Ctrl + F5`

2. **访问测试**：
   - https://soa.laziestlife.com/epet/
   - 检查背景图和宠物图片是否显示

3. **如果图片仍然不显示**：
   - 打开浏览器开发者工具（F12）
   - 查看 Network 标签页
   - 查看图片请求的 URL 和状态码

### 相关文件

- Nginx 配置：`/etc/nginx/conf.d/soa.laziestlife.com.conf`
- 静态文件路径（正确）：`/var/www/iot-ai-doll/frontend/epet/*.png`
- 静态文件路径（构建输出）：`/var/www/iot-ai-doll/frontend/dist/epet/*.png`
- Nginx 错误日志：`/var/log/nginx/error.log`

### 待优化

1. **修复 Nginx 配置**：
   - 找出为什么 `root` 指令没有生效
   - 确保 Nginx 加载正确的配置

2. **自动化部署脚本**：
   - 修改部署脚本，自动将文件复制到 `/var/www/iot-ai-doll/frontend/epet/`
   - 或者在 Nginx 配置中正确使用 `root` 或 `alias`

3. **文件权限**：
   - 确保 Nginx 用户（`www-data`）有读取权限
   - 考虑使用 `chown www-data:www-data` 或修改权限为 `644`

curl -s "https://soa.laziestlife.com/api/epet/9527" | jq

## 🔑 Nginx 配置经验（重要！）

### `root` vs `alias` 指令

location /admin2/ {

## Admin2 HTTPS 404 最终修复 (23:55-23:58)

### 问题
用户无痕模式访问 `https://soa.laziestlife.com/admin2/companions` 仍返回 404，Network 无内容。

### 根因
Nginx 有两套配置同时生效：
1. **sites-available/iot-doll** → `listen 80`, root=`/var/www/iot-ai-doll/frontend`
   - HTTP 请求 `/admin2/` → 找到文件 → 200 OK ✅
2. **conf.d/soa.laziestlife.com.conf** → `listen 443 ssl`, root=`/var/www/iot-ai-doll/frontend/dist`
   - 没有 `/admin2/` location → 找不到文件 → 404 ❌

之前在 conf.d 中添加的 `/admin2/` 配置不知何时丢失（可能被覆盖）。

### 修复
在 `conf.d/soa.laziestlife.com.conf` 的 HTTPS server 块中添加：
```nginx
location /admin2/ {
    root /var/www/iot-ai-doll/frontend;
    try_files $uri $uri/ /admin2/index.html;
    add_header Cache-Control "no-store, no-cache, must-revalidate";
}
```
注意 root 用的是 `/var/www/iot-ai-doll/frontend`（不是 dist），因为 admin2 文件就在这个路径下。

### 结果
- nginx -t ✅ 通过
- systemctl restart nginx ✅ 成功
- curl -skI https://soa.laziestlife.com/admin2/companions → **200 OK**
- Content-Length: 485（新版 v2.0 文件）

### 经验教训
1. Nginx 配置分散在多个文件中（conf.d + sites-enabled），修改时要检查所有位置
2. conf.d 文件可能被覆盖或重写，修改后要验证是否持久生效
3. HTTP 和 HTTPS 可能走不同的 server 块，要分别测试
