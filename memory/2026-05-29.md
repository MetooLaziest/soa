### 21:56 - Nginx 静态文件路径修复

**问题**: 宠物图片显示为破损图标（浏览器无法加载 `pet.png`）

**原因分析**:
- 原 Nginx 配置：`try_files /epet/index.html =404;`
- 这导致所有 `/epet/*` 请求（包括 `pet.png`）都被重定向到 `index.html`
- 浏览器收到的是 HTML 而非 PNG 图片，导致显示失败

**修复方案**:
1. ✅ 修改 Nginx location block：
   ```nginx
   location /epet/ {
       alias /var/www/iot-ai-doll/frontend/dist/epet/;
       try_files $uri $uri/ /epet/index.html;
   }
   ```

2. ✅ 使用 `alias` 指令指向实际目录
3. ✅ 使用 `$uri $uri/` 优先检查静态文件
4. ✅ 只有找不到静态文件时才回退到 `index.html`

**验证结果**:
- ✅ `curl -I https://soa.laziestlife.com/epet/pet.png` 返回 `Content-Type: image/png`
- ✅ 文件大小正确（183KB vs 本地 180KB，差异来自压缩）
- ✅ 图片现在可以正常显示

**经验教训**:
- Nginx 的 `try_files` 会按顺序尝试匹配
- 对于 SPA 应用，需要确保静态资源（图片、CSS、JS）能被正确路由
- 使用 `alias` + `$uri` 可以优先服务静态文件

**当前状态**:

- 饥饿度应该基于 **`last_feed_at`**（上次喂食时间）衰减
- 只有喂食操作才应该重置饥饿度计时器
- 抚摸/铲屎不应该影响饥饿度

- 饥饿度从当前时间重新开始计算，所以显示接近 100

### 21:56 - Nginx 静态文件路径修复

**问题**: 宠物图片显示为破损图标（浏览器无法加载 `pet.png`）

**原因分析**:
- 原 Nginx 配置：`try_files /epet/index.html =404;`
- 这导致所有 `/epet/*` 请求（包括 `pet.png`）都被重定向到 `index.html`
- 浏览器收到的是 HTML 而非 PNG 图片，导致显示失败

**修复方案**:
1. ✅ 修改 Nginx location block：
   ```nginx
   location /epet/ {
       alias /var/www/iot-ai-doll/frontend/dist/epet/;
       try_files $uri $uri/ /epet/index.html;
   }
   ```

2. ✅ 使用 `alias` 指令指向实际目录
3. ✅ 使用 `$uri $uri/` 优先检查静态文件
4. ✅ 只有找不到静态文件时才回退到 `index.html`

**验证结果**:
- ✅ `curl -I https://soa.laziestlife.com/epet/pet.png` 返回 `Content-Type: image/png`
- ✅ 文件大小正确（183KB vs 本地 180KB，差异来自压缩）
- ✅ 图片现在可以正常显示

**经验教训**:
- Nginx 的 `try_files` 会按顺序尝试匹配
- 对于 SPA 应用，需要确保静态资源（图片、CSS、JS）能被正确路由
- 使用 `alias` + `$uri` 可以优先服务静态文件
