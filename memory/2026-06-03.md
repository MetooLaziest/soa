## 晚间补充⑥：user_id=3 赋予宠物 + user_id 跳变分析（21:32）

### 问题现象
- 刷新页面后 user_id 从 2 变成了 3
- 原因：前端 device_id 存 localStorage，但用户可能用了强制刷新/隐私模式导致 localStorage 清空 → 新 device_id → 新 openid → 新用户
- 后端 upsert 逻辑本身没问题（同一 device_id 始终返回同一 user_id）

### user_id=3 宠物数据
- pet_instance_id=11, NFC 333, 海浪沫沫(model 1), 庭院 position=1

- pet_instance_id=13, NFC 10011, 糖心莓莓(model 3), 收藏中

### 发现的额外问题
1. **NFC ID 全局唯一约束** —— 同一个 NFC ID 不能给不同用户，需要用不同的 NFC

3. **后端 openid 拼接问题** —— 后端代码把 device_id 再包了一层 `guest_` 前缀变成 `guest_guest_xxx`，虽然不影响功能但不优雅

## 晚间补充⑫：修复藏品库图片映射 + yard/add 500 错误（21:59）

### 问题1：宠物图片映射错误
- **现象**：团子糯糯显示糖心莓莓的图，糖心莓莓显示团子糯糯的图
- **根因**：`App.tsx` 中 `PET_IMAGES` 映射表 model 2/3 的图片路径写反了
- **修复**：
  - model 2（团子糯糯）：`pet-pink.png` → `pet-idle.png`
  - model 3（糖心莓莓）：`pet-idle.png` → `pet-pink.png`

### 问题2：POST /pet/yard/add 返回 500
- **现象**：设置糖心莓莓时 `POST /pet/yard/add failed: 500`
- **根因**：后端 `routes/pet.js` 中 `yard/add` 路由缺少 `await client.query('BEGIN')`，直接执行 INSERT ON CONFLICT DO UPDATE，事务状态不一致导致崩溃
- **修复**：
  - 补上完整事务流程：BEGIN → 检查数量 → 检查是否已存在 → INSERT 或 UPDATE → COMMIT
  - 对已存在但 is_active=false 的记录走 UPDATE 重新激活

### 关键经验
- **前端 PET_IMAGES 映射表必须和实际图片内容一致**，model_id → 图片 URL 不能凭直觉猜测
- **PostgreSQL 事务必须完整**：BEGIN + 操作 + COMMIT/ROLLBACK，缺一不可，否则并发或冲突场景会 500
- 后端 pet.js 已重写并部署到服务器 `/var/www/epet1-server/routes/pets.js`

### 当前状态
- ✅ 图片映射已修正
- ✅ yard/add 500 已修复
- ⏳ 需用户验证两个问题是否都解决
