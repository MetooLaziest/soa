# 13:08 补充 - E-Pet v6.6 字段名修复

## Bug 根因
- **错误信息**：`Cannot read properties of undefined (reading 'last_feed_at')`

  - 后端 API 返回**全驼峰**：`petId`, `fullness`, `totalFeeds`, `totalPoops`, `monsterType`, `lastInteractAt`

- **影响范围**：喂食、抚摸、铲屎三个操作全部报错（因为都依赖 `petId` 来构造 API URL 和更新数组）

## 修复内容
- 全局替换所有 `p.pet_id` → `p.petId`
- 全局替换所有 `visiblePets[i].pet_id` → `visiblePets[i].petId`
- 涉及函数：`renderPetCards()`, `onPetClick()`, `doFeed()`, `doPet()`, `doClean()`, `getPoopCount()` 中间变量
- 同时确保 `monsters` 数组中的 `m.petId` 保持一致（monsters 本身就是驼峰）

## 部署
- 前端: `/var/www/iot-ai-doll/frontend/dist/epet/index.html` (v6.6)

## 经验教训
- **前后端字段名必须统一约定**：后端返回驼峰，前端也必须用驼峰
- **API 返回数据结构是调试第一优先级**：报 `undefined.xxx` 时先检查字段名是否匹配

## 性能优化方案（15000 条 range 查询）

### 推荐方案：nfc_to_model 枚举表（O(1) 主键查询）

```sql
-- 预计算 NFC ID → model_id 的直接映射
CREATE TABLE nfc_to_model (
  nfc VARCHAR(20) PRIMARY KEY,  -- 00001, 00002, ... 15000（补零对齐）
  model_id INTEGER REFERENCES models(model_id)
);

-- 初始化：从 model_id_ranges 展开生成 15000 条记录
INSERT INTO nfc_to_model (nfc, model_id)
SELECT 
  LPAD(gs.n::text, 5, '0'),  -- 1→00001, 2→00002, ..., 15000→15000
  m.model_id
FROM generate_series(1, 15000) AS gs(n)
JOIN model_id_ranges mr ON gs.n BETWEEN mr.start_id AND mr.end_id
JOIN models m ON m.model_id = mr.model_id;

-- 查询变成 O(1) 主键查找
SELECT model_id FROM nfc_to_model WHERE nfc = '07345';
```

**优点**：
- 查询：O(1) 主键扫描，无索引失效问题
- 数据量极小（15000 行，每行约 20 字节 ≈ 300KB，可全表缓存）
- 维护简单：`model_id_ranges` 变更时重建 `nfc_to_model`（不频繁）
- 无需改业务代码逻辑

**备选方案：BRIN 索引**（若不想维护枚举表）
```sql
CREATE INDEX idx_ranges_brin ON model_id_ranges USING BRIN (start_id, end_id);
```
BRIN 对有序范围扫描效率高，适合这种一次性初始化的静态数据。

**结论**：推荐 `nfc_to_model` 枚举表方案。

# 16:29 — 修复 auto-refresh 与操作竞态导致数据延迟（v6.11）

## Bug 描述
喂食/铲屎后不手动刷新，饱腹感和排泄物图标要等很久才更新。

## 根因
`loadVisiblePets()` 每15秒执行一次时，会 `visiblePets = []` 清空数组，然后重新 fetch。
如果此时喂食/铲屎的异步 fetch 还在进行中，API 返回新数据时 `visiblePets` 已经被清空 → `for` 循环里的 `break` 永远不执行 → 数组中该宠物数据还是旧的 → UI 不更新。

## 修复
`loadVisiblePets()` 改为**合并更新**（merge），不再清空数组：
```javascript
// 修复前：visiblePets = []; visiblePets.push(d.pet);
// 修复后：用 findIndex 找到对应宠物，用 Object.assign 合并
var idx = visiblePets.findIndex(function(p){ return p.petId === d.pet.petId; });
if (idx >= 0) {
  visiblePets[idx] = Object.assign({}, visiblePets[idx], d.pet);
} else {
  visiblePets.push(d.pet);
}
```
同时 `doFeed`/`doClean` 也用 `Object.assign` 合并，兼容此逻辑。
