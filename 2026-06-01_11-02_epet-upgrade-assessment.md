# E-Pet 升级项目评估报告

**评估时间**: 2026-06-01 11:02  
**评估对象**: E-Pet 电子宠物与真实玩偶互动升级

---

## 一、项目概述

### 当前架构
- **前端**: H5 PWA (HTML/CSS/JS)，已部署于 `https://soa.laziestlife.com/epet/`
- **后端**: Node.js + Express + PostgreSQL
- **数据库**: 
  - `epet` 数据库: `pets` 表（宠物状态数据）
  - `iot_doll` 数据库: `profiles` 表（用户认证）、`companion_models` 表（AI人设）
- **现有功能**: 喂食、抚摸、铲屎、饱腹感计算、睡眠系统

### 目标升级内容
1. NFC绑定用户与藏品
2. 藏品碎片系统
3. 用户藏品库查看
4. 对话体验（充值/喂墩墩兽）
5. 墩墩兽碎片合成
6. 藏品切换
7. 梦界旅游故事（旅行青蛙模式）

## 二、功能模块拆解与开发周期评估

### Phase 1: 基础设施（2-3周）

| 模块 | 工作量 | 说明 |
|------|--------|------|
| 用户认证系统 | 3-5天 | 已有 `profiles` 表，需扩展登录态管理 |
| NFC绑定逻辑 | 5-7天 | 新表设计：`nfc_bindings`、`user_collections` |
| 藏品模型设计 | 3-5天 | 藏品表、碎片表、合成规则表 |
| API权限控制 | 2-3天 | 区分"本人"vs"他人"访问权限 |

**关键表结构建议**:
```sql
-- 藏品定义表
CREATE TABLE collectibles (
    id UUID PRIMARY KEY,
    model_id VARCHAR(50),        -- 玩偶型号（如 MOMOTOY-V1）
    name VARCHAR(100),
    description TEXT,
    image_url VARCHAR(255),
    rarity INTEGER,              -- 稀有度 1-5
    fragment_count INTEGER,      -- 合成所需碎片数
    personality_config JSONB     -- 性格设定（关联 companion_models）
);

-- 用户藏品表
CREATE TABLE user_collections (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES profiles(id),
    collectible_id UUID REFERENCES collectibles(id),
    nfc_id VARCHAR(100),         -- NFC芯片ID
    acquired_at TIMESTAMP,
    fragment_count INTEGER DEFAULT 0,  -- 当前碎片数
    is_primary BOOLEAN DEFAULT false,  -- 是否主藏品
    UNIQUE(user_id, nfc_id)
);

-- 碎片记录表
CREATE TABLE user_fragments (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES profiles(id),
    collectible_id UUID REFERENCES collectibles(id),
    source_nfc_id VARCHAR(100),  -- 来源NFC（重复触碰获得）
    obtained_at TIMESTAMP
);

-- 合成记录表
CREATE TABLE synthesis_history (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES profiles(id),
    from_fragments INTEGER,
    to_collectible_id UUID REFERENCES collectibles(id),
    synthesized_at TIMESTAMP
);
```

### Phase 2: 核心玩法（3-4周）

| 功能 | 工作量 | 技术要点 |
|------|--------|----------|
| **NFC触碰流程** | 5-7天 | 首次绑定/重复碎片/他人查看 |
| **藏品库展示** | 3-5天 | 网格展示、切换交互 |
| **碎片合成** | 3-5天 | 合成动画、规则校验 |
| **充值系统** | 5-7天 | 微信支付/支付宝对接 |

**NFC触碰状态机**:
```
NFC触碰 → 读取NFC_ID
    ↓
检查NFC是否已绑定藏品
    ↓
    ├── 未绑定 → 首次绑定流程
    │            ├── 已登录 → 绑定到当前用户
    │            └── 未登录 → 跳转登录 → 再绑定
    │
    └── 已绑定 → 检查当前用户
                 ├── 本人 → 打开交互页
                 ├── 他人 → 展示归属+藏品库
                 └── 未登录 → 展示归属+引导登录
```

### Phase 3: 高级功能（4-6周）

| 功能 | 工作量 | 技术要点 |
|------|--------|----------|
| **对话体验** | 7-10天 | AI对话集成、记忆系统、充值扣费 |
| **梦界旅游** | 7-10天 | 旅行青蛙模式、时间计算、带回物品 |
| **具身智能迁移** | 10-14天 | 记忆导出/导入、跨设备同步 |

**对话体验架构**:
```
用户输入 → 检查余额/墩墩兽
    ↓
调用AI API（已有阿里千问接入）
    ↓
存储对话历史 → 更新记忆向量
    ↓
扣费（充值余额或消耗墩墩兽）
    ↓
返回回复
```

**梦界旅游机制**:
- 触发条件: 宠物状态良好（饱腹感>50）+ 用户主动派遣
- 旅行时长: 2-24小时（随机或用户选择）
- 带回物品: 碎片、道具、IP联名纪念品（软广位）
- 明信片系统: 旅行途中发送照片/故事卡片

## 五、技术风险与建议

### 高风险项
1. **支付合规**: 虚拟商品充值需申请ICP证或游戏版号
   - 建议: 咨询法务，考虑"会员服务"模式规避
   
2. **NFC兼容**: 不同手机NFC读取稳定性差异大
   - 建议: 提供二维码备用方案

3. **AI记忆一致性**: 多设备/多会话间记忆同步复杂
   - 建议: Phase 3再实现，MVP用简单上下文

### 中风险项
1. **并发性能**: 高并发NFC触碰可能数据库瓶颈
   - 建议: 加Redis缓存层

2. **数据安全**: 用户藏品数据需防刷防篡改
   - 建议: 关键操作加签名验证
